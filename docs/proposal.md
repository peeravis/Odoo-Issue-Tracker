## บทที่ 1 บทนำ

### 1.1 ที่มาและความสำคัญของปัญหา

ในองค์กรที่ให้บริการด้านการนำระบบสารสนเทศไปใช้งาน (System/Software Implementation) ให้กับลูกค้าหรือหน่วยงานหลายราย เช่น การวางระบบ ERP, การปรับแต่งซอฟต์แวร์ตามความต้องการเฉพาะ (customization) หรือการติดตั้งระบบให้ลูกค้าองค์กรจำนวนมาก แต่ละโครงการมักถูกดำเนินการโดยทีมงานที่แตกต่างกัน และในทางปฏิบัติแต่ละทีมมักเลือกใช้เครื่องมือติดตามงาน (Issue/Task Tracking) ที่แตกต่างกันไป หรือแม้จะใช้เครื่องมือเดียวกัน (เช่น Jira, Trello, Excel) แต่ไม่มีการกำหนดโครงสร้างข้อมูลหรือขั้นตอนการทำงานที่เป็นมาตรฐานร่วมกัน

ผลที่ตามมาคือ:

1. **ข้อมูลไม่เป็นมาตรฐานเดียวกัน (Data Inconsistency)** — แต่ละโครงการบันทึกสถานะ ระดับความสำคัญ หรือรายละเอียดการแก้ไขปัญหา (solution) ด้วยรูปแบบที่ต่างกัน ทำให้ผู้บริหารไม่สามารถเปรียบเทียบหรือรวบรวมภาพรวมข้ามโครงการได้
2. **ขาดการควบคุมคุณภาพกระบวนการ (Process Governance)** — ไม่มีการบังคับให้ทุกโครงการต้องผ่านขั้นตอนเดียวกัน เช่น การระบุสาเหตุ (root cause), การลงนามยืนยันจากลูกค้า (sign-off), หรือการบันทึกประวัติการเปลี่ยนแปลง (activity log) ทำให้ตรวจสอบย้อนหลัง (traceability) ได้ยาก
3. **ต้นทุนการบริหารจัดการเครื่องมือสูง** — การใช้เครื่องมือสำเร็จรูปทั่วไป (Jira, ClickUp, Redmine) ต้องมีการตั้งค่า (configuration) ใหม่ทุกครั้งที่เริ่มโครงการ และไม่ได้ออกแบบมาเฉพาะสำหรับ workflow ของงาน Implementation โดยตรง (เช่น ไม่มีแนวคิดเรื่อง "solution field" ที่ผูกกับลูกค้าและ sign-off โดยเฉพาะ)

งานวิจัยนี้จึงมุ่งพัฒนาแพลตฟอร์มกลางที่ให้ทุกโครงการภายในองค์กรใช้งานร่วมกัน (single platform, multi-project) โดยกำหนด **มาตรฐานเวิร์กโฟลว์กลาง (Standardized Implementation Workflow)** ที่บังคับใช้เหมือนกันทุกโครงการ ทั้งในด้านโครงสร้างข้อมูล (master data), สถานะงาน, การมอบหมายงาน, และการบันทึกประวัติ เพื่อให้องค์กรสามารถบริหารจัดการ ติดตาม และวิเคราะห์ภาพรวมงาน Implementation ของทุกโครงการได้จากจุดเดียว

### 1.2 วัตถุประสงค์ของการวิจัย

1. เพื่อออกแบบและพัฒนาแพลตฟอร์มติดตามงาน Implementation แบบรวมศูนย์ ที่รองรับการทำงานพร้อมกันของหลายโครงการบนมาตรฐานเวิร์กโฟลว์เดียวกัน
2. เพื่อกำหนดและนำมาตรฐานกลาง (standardized master data, สถานะงาน, ฟิลด์บันทึกการแก้ไขปัญหา, กลไก sign-off และ activity log) มาบังคับใช้ในทุกโครงการผ่านสถาปัตยกรรมของระบบ
3. เพื่อประเมินประสิทธิภาพและความสามารถในการใช้งาน (usability) ของแพลตฟอร์มที่พัฒนาขึ้น เปรียบเทียบกับรูปแบบการทำงานเดิมที่แยกเครื่องมือ/มาตรฐานตามแต่ละโครงการ

### 1.3 ขอบเขตของการวิจัย

- พัฒนาเป็นเว็บแอปพลิเคชันด้วย Next.js (App Router) + PostgreSQL + Prisma ตามสถาปัตยกรรมต้นแบบที่มีอยู่แล้ว
- ขอบเขตการใช้งานอยู่ภายในองค์กรเดียว (single organization) ที่บริหารจัดการหลายโครงการ (multi-project) ไม่ใช่ SaaS หลายองค์กร (multi-tenant)
- ครอบคลุมกระบวนการหลักของงาน Implementation ได้แก่ การบันทึกปัญหา/งาน (issue) การมอบหมายผู้รับผิดชอบ การกำหนดระดับความสำคัญและกำหนดส่ง การบันทึกวิธีแก้ไขและปิดงาน และการบันทึกประวัติการเปลี่ยนแปลง (audit trail)
- ไม่รวมส่วนของการเงิน/billing และการบริหารสัญญาลูกค้า (CRM) โดยตรง
- การประเมินผลใช้กลุ่มตัวอย่างผู้ใช้งานจริงหรือผู้เชี่ยวชาญ (expert/user evaluation) จำนวน _[ระบุจำนวน]_ คน จากทีมที่เกี่ยวข้องกับงาน implementation หลายโครงการ

### 1.4 ประโยชน์ที่คาดว่าจะได้รับ

1. องค์กรมีแพลตฟอร์มกลางที่ใช้ติดตามงาน Implementation ได้ทุกโครงการ ลดความซ้ำซ้อนในการจัดหา/ตั้งค่าเครื่องมือ
2. ผู้บริหารสามารถวิเคราะห์และเปรียบเทียบภาพรวมงานข้ามโครงการได้ เนื่องจากข้อมูลอยู่ในมาตรฐานเดียวกัน
3. เพิ่มความสามารถในการตรวจสอบย้อนหลัง (traceability) และความรับผิดชอบ (accountability) ผ่านกลไก activity log ที่บังคับใช้เป็นมาตรฐาน
4. เป็นต้นแบบ (reference model) สำหรับองค์กรลักษณะเดียวกันที่ต้องการมาตรฐานกลางในการบริหารโครงการ implementation หลายโครงการพร้อมกัน

## บทที่ 2 ทบทวนวรรณกรรมและงานวิจัยที่เกี่ยวข้อง

### 2.1 แนวคิดและทฤษฎีที่เกี่ยวข้อง

- **Issue/Task Tracking Systems** — แนวคิดพื้นฐานของระบบติดตามปัญหาซอฟต์แวร์ (bug/issue tracking) และองค์ประกอบสำคัญ เช่น สถานะ (status), ผู้รับผิดชอบ (assignee), ความสำคัญ (priority)
- **IT Service Management (ITIL) — Problem & Change Management** — แนวคิดเรื่องมาตรฐานกระบวนการจัดการปัญหาและการเปลี่ยนแปลงที่ใช้อ้างอิงในการออกแบบ workflow มาตรฐาน
- **Software Process Standardization / CMMI** — แนวคิดเรื่องการกำหนดมาตรฐานกระบวนการทำงานเพื่อควบคุมคุณภาพและความสม่ำเสมอของผลลัพธ์ข้ามหลายโครงการ
- **Multi-Project Management** — แนวคิดการบริหารจัดการหลายโครงการพร้อมกันภายใต้องค์กรเดียว (Program/Portfolio Management)

### 2.2 งานวิจัยและเครื่องมือที่เกี่ยวข้อง

เปรียบเทียบแพลตฟอร์มที่มีอยู่ในปัจจุบัน เพื่อชี้ให้เห็นช่องว่างของงานวิจัย:

| เครื่องมือ | จุดเด่น | ข้อจำกัดเมื่อใช้กับงาน Implementation หลายโครงการ |
|---|---|---|
| Jira | ยืดหยุ่นสูง, ecosystem ใหญ่ | ต้องตั้งค่า workflow เองทุกโปรเจกต์ ไม่มีมาตรฐานบังคับ ต้นทุนสูงเมื่อใช้จำนวนมาก |
| Redmine / ClickUp | Open-source / ปรับแต่งได้ | ไม่มีแนวคิด "solution field" หรือ sign-off เฉพาะงาน implementation |
| Excel / Google Sheet | ใช้งานง่าย ต้นทุนต่ำ | ไม่มีการบังคับมาตรฐาน ไม่มี audit trail ไม่รองรับการวิเคราะห์ข้ามโครงการ |
| ระบบที่พัฒนาขึ้น (นำเสนอ) | มาตรฐานกลางถูกฝังในสถาปัตยกรรมระบบ (master data ส่วนกลาง, activity log บังคับ, solution field มาตรฐาน) | ยืดหยุ่นน้อยกว่าเครื่องมือสำเร็จรูปทั่วไป (trade-off ที่ยอมรับได้เพื่อแลกกับมาตรฐาน) |

_[หมายเหตุ: ส่วนนี้ต้องขยายด้วยการอ้างอิงบทความวิชาการ/งานวิจัยที่เกี่ยวข้องจริง (citation) เพิ่มเติมก่อนใช้ยื่นเสนออาจารย์ที่ปรึกษาอย่างเป็นทางการ]_

### 2.3 ช่องว่างของงานวิจัย (Research Gap)

งานวิจัยและเครื่องมือที่มีอยู่ส่วนใหญ่มุ่งเน้นความยืดหยุ่นให้แต่ละทีม/โครงการปรับแต่งได้อิสระ ซึ่งขัดแย้งกับความต้องการขององค์กรที่ต้องการ**มาตรฐานกลางที่บังคับใช้เหมือนกันทุกโครงการ**เพื่อให้สามารถเปรียบเทียบและวิเคราะห์ภาพรวมได้ งานวิจัยนี้จึงเสนอแนวทางออกแบบสถาปัตยกรรมระบบที่ฝังมาตรฐาน (standard-by-design) ไว้ในระดับโครงสร้างข้อมูลและกระบวนการ แทนที่จะปล่อยให้เป็นข้อตกลงเชิงนโยบายที่ไม่มีการบังคับใช้จริงในทางเทคนิค

## บทที่ 3 วิธีดำเนินการวิจัย

### 3.1 กรอบแนวคิดและสถาปัตยกรรมระบบ

ระบบต้นแบบที่พัฒนาขึ้นแล้วบางส่วนประกอบด้วยองค์ประกอบหลักที่สนับสนุนแนวคิดมาตรฐานกลาง ได้แก่:

- **Master Data ส่วนกลาง** — dropdown/ค่ามาตรฐาน (เช่น สถานะ, ระดับความสำคัญ) ถูกกำหนดจากส่วนกลางและใช้ร่วมกันทุกโครงการ แทนที่แต่ละโครงการจะกำหนดเอง
- **โครงสร้าง Issue มาตรฐาน** — ทุก issue ในทุกโครงการมีโครงสร้างฟิลด์เดียวกัน (status, priority, assignee, due date, solution field) ทำให้เปรียบเทียบข้ามโครงการได้โดยตรง
- **Activity Log บังคับ** — การเปลี่ยนแปลงทุกฟิลด์ของ issue ถูกบันทึกอัตโนมัติ เป็นกลไกมาตรฐานด้าน traceability ที่ไม่ขึ้นกับดุลยพินิจของผู้ใช้แต่ละโครงการ
- **Dashboard ภาพรวมข้ามโครงการ** — สรุปสถิติแยกตามโครงการและผู้รับผิดชอบบนฐานข้อมูลเดียวกัน

ส่วนที่จะพัฒนาเพิ่มเติมสำหรับงานวิจัยนี้ (เพื่อยกระดับจาก "ระบบต้นแบบ" ไปสู่ "งานวิจัยเชิงมาตรฐาน"):

1. **Implementation Workflow Model** — กำหนดขั้นตอนมาตรฐานของงาน implementation อย่างเป็นทางการ (เช่น Kickoff → Requirement → Configuration → UAT → Sign-off → Go-live → Support) และบังคับใช้ผ่านระบบ แทนสถานะอิสระที่ผู้ใช้กำหนดเอง
2. **Compliance/Conformance Checking** — กลไกตรวจสอบว่าแต่ละโครงการปฏิบัติตามมาตรฐานกลางครบถ้วนหรือไม่ (เช่น ฟิลด์บังคับก่อนปิดงาน, sign-off ที่ขาดหาย)
3. **Cross-Project Standardization Metrics** — ตัวชี้วัดเชิงปริมาณที่ใช้ประเมินระดับความสม่ำเสมอของข้อมูลระหว่างโครงการ (data consistency score) เพื่อใช้เป็นผลการทดลอง

### 3.2 เครื่องมือและเทคโนโลยีที่ใช้พัฒนา

- Frontend/Backend: Next.js 16 (App Router), TypeScript, Tailwind CSS
- Database: PostgreSQL, Prisma ORM (Driver Adapter)
- Authentication: Custom JWT session (jose)
- อื่น ๆ: ExcelJS (นำเข้า/ส่งออกข้อมูล)

### 3.3 ขั้นตอนการดำเนินการวิจัย

1. ทบทวนวรรณกรรมและวิเคราะห์เครื่องมือที่เกี่ยวข้อง
2. วิเคราะห์และออกแบบ Implementation Workflow Model มาตรฐาน (ร่วมกับผู้เชี่ยวชาญ/ผู้ใช้งานจริงในองค์กร)
3. ออกแบบฐานข้อมูลและสถาปัตยกรรมระบบเพิ่มเติมจากต้นแบบเดิม
4. พัฒนาแพลตฟอร์มตามมาตรฐานที่กำหนด (Agile/Iterative development)
5. ทดสอบระบบกับกลุ่มตัวอย่างผู้ใช้งานจากหลายโครงการ
6. ประเมินผลด้วยแบบสอบถามความพึงพอใจ/usability testing และตัวชี้วัด data consistency score
7. สรุปผลและจัดทำรายงานวิทยานิพนธ์

### 3.4 การประเมินผล

- **Usability Evaluation**: แบบสอบถามมาตรฐาน เช่น System Usability Scale (SUS) จากผู้ใช้งานจริงหลายโครงการ
- **Standardization Effectiveness**: เปรียบเทียบความสม่ำเสมอของข้อมูล (เช่น อัตราการกรอกฟิลด์ solution ครบถ้วน, อัตรา sign-off ที่บันทึกครบ) ระหว่างช่วงก่อน-หลังใช้แพลตฟอร์ม
- **Expert Review**: ให้ผู้เชี่ยวชาญด้านการบริหารโครงการ/PM ประเมินความเหมาะสมของ workflow มาตรฐานที่ออกแบบ
